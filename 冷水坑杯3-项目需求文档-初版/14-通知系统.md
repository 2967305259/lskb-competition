# 通知系统

## 通知数据模型

```python
class Notification(db.Model):
    id         # 主键
    user_id    # FK → users.id，索引
    title      # 通知标题
    message    # 通知内容
    read       # 是否已读（默认 False）
    created_at # 创建时间
```

## 通知触发点

系统在以下场景自动创建通知：

| 触发场景 | 接收者 | 标题示例 |
|---|---|---|
| 队长邀请选手 | 被邀请选手 | 您收到了来自队伍 XXX 的邀请 |
| 选手接受邀请 | 队长 | 选手 XXX 已加入您的队伍 |
| 选手拒绝邀请 | 队长 | 选手 XXX 拒绝了您的邀请 |
| 自动匹配完成 | 双方选手 | 您有一场新的比赛 |
| 选手提交比赛时间 | 对手 | 对手提议了比赛时间 |
| 选手确认时间 | 对手 | 对手已确认比赛时间 |
| 比赛结果确定 | 双方选手 | 比赛结果已确定 |
| 比赛出现争议 | 双方选手 | 比赛出现争议 |
| 管理员裁决争议 | 双方选手 | 管理员已解决比赛争议 |
| 选手弃权 | 对手 | 对手已弃权 |
| 管理员修改结果 | 双方选手 | 管理员修改了比赛结果 |
| 管理员审核比赛 | 双方选手 | 管理员审核了您的比赛 |
| 用户审批通过 | 用户 | 您的账号已通过审批 |
| 用户审批拒绝 | 用户 | 您的账号未通过审批 |

## 当前实现状态

### 已实现
- 通知创建：使用 `create_notification(user_id, title, message)` 辅助函数
- 通知角标：`unread_notifications_count` 通过上下文处理器注入所有模板
- 导航栏铃铛图标：显示未读数量

### ⚠️ 未实现
- **通知列表页面**：没有 `/notifications` 路由来查看所有通知
- **标记已读**：没有 API 或路由来标记通知为已读
- **通知详情**：没有单条通知的查看页面
- **全部标记已读**：没有批量操作
- **通知筛选**：无法按类型、时间筛选
- **通知删除**：无法删除旧通知
- **通知推送**：仅有数据库存储，无 WebSocket/SSE 实时推送

## 上下文注入

```python
@app.context_processor
def inject_notifications():
    if current_user.is_authenticated:
        count = Notification.query.filter_by(
            user_id=current_user.id, read=False
        ).count()
        return {'unread_notifications_count': count}
    return {'unread_notifications_count': 0}
```

## 通知系统改进建议

### 短期（必须实现）
1. **通知列表页**：`GET /notifications` - 分页展示所有通知
2. **标记已读接口**：`POST /notifications/<id>/read`
3. **全部标记已读**：`POST /notifications/read-all`

### 中期（功能增强）
4. **通知分类**：添加 `notification_type` 字段（invitation/match/result/system 等）
5. **通知筛选**：按类型、已读/未读、时间范围筛选
6. **通知删除**：支持单条删除和批量删除
7. **通知偏好设置**：用户可选择接收哪些类型的通知

### 长期（体验优化）
8. **实时推送**：WebSocket 或 SSE 实时推送新通知
9. **邮件通知**：重要通知（比赛匹配、争议）可发送邮件
10. **通知有效期**：旧通知自动归档/清理机制
