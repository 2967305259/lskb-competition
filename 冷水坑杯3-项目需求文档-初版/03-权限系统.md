# 权限系统

## 角色体系

系统采用四级角色层级结构：

| 角色 | 标识 | 权限范围 |
|---|---|---|
| **超级管理员** (super_admin) | `role='super_admin'` | 全部权限 + 管理其他管理员 |
| **管理员** (admin) | `role='admin'` | 赛事管理、用户审批、数据处理 |
| **队长** (captain) | `role='captain'` | 创建/管理队伍、邀请队员、参加比赛 |
| **选手** (player) | `role='player'` | 加入队伍、选择肉鸽、参加比赛 |

**默认账号**：
- `sahu` — 超级管理员（密码见部署文档）
- `admin` / `admin123` — 管理员（首次登录强制修改密码）

## 注册与审批流程

```
用户注册 → approval_status='pending' → 管理员审批
                                           ├─ approve → 可以登录使用
                                           └─ reject → 无法使用系统
```

- 注册时所有人默认为 `player` 角色
- 注册受 `registration_open` 开关控制
- 审批通过后发送通知

## 角色升级路径

```
player → promote → captain → promote → admin → promote (super_admin only) → super_admin
          ← demote ←          ← demote ←          ← demote (super_admin only) ←
```

- 降级队长前需确认其没有队伍
- 超级管理员的升降级仅超级管理员可操作

## 路由级别权限控制

| 蓝图 | 权限要求 |
|---|---|
| `main_bp` (`/`) | 公开访问（首页、队伍列表、赛事大厅） |
| `auth_bp` (`/auth`) | 公开访问（登录、注册） |
| `player_bp` (`/player`) | `@login_required` + 角色检查 |
| `captain_bp` (`/captain`) | `@login_required` + `role='captain'` |
| `admin_bp` (`/admin`) | `@login_required` + `role in ('admin', 'super_admin')` |
| `tournament_bp` (`/tournament`) | 部分公开（hall/rankings/results），部分需登录 |

## 权限实现方式

当前使用 **装饰器模式** + 视图内角色检查：

```python
# 示例：admin 蓝图所有路由检查管理员角色
@admin_bp.before_request
def check_admin():
    if not current_user.is_authenticated:
        return redirect(url_for('auth.login'))
    if not (current_user.is_admin() or current_user.is_super_admin()):
        abort(403)
```

## User 模型关键方法
- `is_super_admin()` — 检查是否为超级管理员
- `is_admin()` — 检查是否为管理员（含 super_admin）
- `is_captain()` — 检查是否为队长
- `is_player()` — 检查是否为选手
- `is_approved()` — 检查是否已通过审批
- `has_team()` — 检查是否已有队伍

## 权限系统不足与改进建议

1. **缺少细粒度权限**：当前仅按角色划分，无操作级别的权限控制（如某些管理员只能管理队伍不能管理比赛）
2. **无权限变更日志**：角色升降级操作未记录到 SystemLog
3. **CSRF 保护**：已启用 Flask-WTF CSRF，但部分 POST 路由缺少显式保护
4. **会话安全**：默认24小时会话超时，建议生产环境缩短至2-4小时
5. **密码策略**：无密码复杂度要求，建议增加最小长度和复杂度规则
6. **无操作日志审计**：SystemLog 模型存在但未使用，管理员关键操作无审计追踪
